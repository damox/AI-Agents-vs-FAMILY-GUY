# AI-Agents-vs-FAMILY-GUY

### A multi-agent system that imitates the “Family Guy writing process” as depicted in the South Park parody.

//  Built during the Google & Kaggle 5-Day AI Agents Intensive Bootcamp (2025). To view Kaggle Notebook: https://www.kaggle.com/code/damlapart/ai-agents-vs-family-guy/


### Problem Statement:
This project is inspired by a famous South Park gag that satirizes Family Guy’s signature flashback jokes. In Family Guy, Peter Griffin constantly jumps into absurd, rapid-fire memories involving random celebrities, strange objects, or bizarre settings. The humor comes from the surreal disconnect between the main scene and the flashback — and the chaotic logic used to produce it.

In South Park’s parody, these flashback jokes are shown to be generated by manatees selecting “idea balls” in an aquarium. Each ball contains a random noun, verb, place, or famous person. When five balls are selected, the writers must build the next joke using exactly those items. The joke lands not because it makes sense, but because it barely does — yet still fits the Family Guy formula. To watch that scene:https://www.youtube.com/watch?v=qTC9j0QpCBM

This raised a question:
What if we recreate that exact pipeline — the manatee-style “idea ball system” — using a multi-agent AI architecture?

The challenge becomes threefold:

* Can a sequence of agents generate a flashback scenario using the same random-item pipeline mocked in South Park?

* Can another agent, acting as an “investigator,” check whether Family Guy has already created an identical or extremely similar flashback joke?

* And ultimately, can we ever get a “JACKPOT”? //meaning: the generated scenario already exists in a real Family Guy episode.

This transforms a piece of TV satire into a creative AI experiment: a humorous test of whether a multi-agent reasoning system can accidentally (or statistically) replicate the chaotic randomness and comedic writing style of Family Guy." not replication, but there is a very low possibility to get the same because it's mathematically kinda infinity possibilities-- like winning the lottery.

### Why agents?
The first two steps of the pipeline — generating random “idea balls” and selecting five of them — could technically be done with simple scripts, even before GPT-3. These parts don’t require real reasoning "technically".

But the third agent, the scenario-writer, does.
It must take five unrelated items and turn them into a coherent flashback. That requires a contextual understanding and flexible reasoning — something far beyond what simple code can achieve.

The fourth agent, the investigator, is even more demanding.
To work properly, it would usually require a full machine-learning project: collecting and cleaning all Family Guy transcripts, embedding and indexing them, running similarity searches. Building that system manually could take weeks or months. (For now, it is still not working accordingly, but stated in the future work part.) Using a multi-agent architecture enables this to become a usable workflow within days, with each agent handling a specific part of the process.

And overall, I wanted to mimic the writing approach shown in that South Park parody scene by building this multi-agent sequential pipeline as a collaborative “team,” because — at least for public use — agents are currently the closest things we have to a primitive AGI.

#### Sequential Agent Pipeline:

The system is organized as a Sequential Agent Pipeline, where each agent performs a distinct role, passes its output to the next, and writes to specific keys:

Agents: 4 agents + 1 root agent

* #### idea_ball_agent:

Generates 100 random items drawn from verbs, nouns, pop culture figures, and places.

Output: writes all generated items to {generated_ideaballs}.

* #### manatee_agent:

Randomly selects 5 items from {generated_ideaballs}, mimicking the manatees selecting idea balls in the South Park parody.

Output: writes selected items to {picked_ideaballs}.

* #### scenario_writer_agent:

Constructs two reasonable sentences using strictly the items in {picked_ideaballs}.

Does not attempt humor; loosely follows the “Peter Griffin flashback” narrative as a style example.

Uses temperature and top_p parameters to reduce deterministic output.

Output: writes the resulting scenario to {generated_scenario}.

* #### inspector_agent:

Compares strictly the {generated_scenario} against existing flashback content online (using Google Search as a tool).

Checks for exact matches, finds closest similar examples, or returns “Better luck next time” if below threshold.

Output: writes the comparison result to {comparison_result}.

* #### root_agent:

Orchestrates the sequential workflow, passing outputs between agents according to the keys above, and returns the final result to the user.

### If I had more time, this is what I'd do:
* Structured Family Guy Flashback Database:
A comprehensive repository of all flashback moments, with metadata tagging for nouns, verbs, locations, and celebrity names.

* Multi-Agent Search & Comparison Pipeline:
To improve the inspection of whether a “Jackpot” occurs or if similar sketches already exist:

* Transcript Collector Agent: gathers and curates all episode transcripts.

* Vectorization & Projection Agent: converts transcripts into a searchable embedding space.

* Comparison Agent: compares generated scenarios against the corpus using nearest-neighbor search to find exact or closest matches.

* Gamification Approach:
The system could be automated to run each week just before a new episode airs, generating scenarios up to the point of the episode. After the episode is released, the Inspector Agent evaluates how similar the generated scenarios were to the actual content, scoring them out of 100%. User responses could be collected before and after each episode to enhance engagement and make "people" act as "people" again…
